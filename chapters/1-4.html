什么是规则系统

核心原理

人类编写规则 → 机器执行规则 → 得出结果

就像一本操作手册：遇到情况A，就执行操作B。

// 规则系统的代码示例

if 邮件 contains "免费" and 邮件 contains "点击"
  then 标记为垃圾邮件

if 温度 > 30 and 湿度 > 80
  then 发出高温预警

if 订单金额 > 10000
  then 需要主管审批

例子：电商平台的规则引擎

if 新用户注册 then 发送欢迎邮件 + 设置新手标识
if 订单金额 > 99 then 免运费
if 用户等级 = VIP and 购买次数 > 10 then 专属折扣15%
if 退货率 > 30% then 加入黑名单

规则系统的优缺点

维度✅ 优点❌ 缺点
可解释性 规则是人写的，可以检查 规则多时难以维护
稳定性 结果可预测 无法处理边界情况
调试 出问题可以逐条检查 规则冲突时难排查
学习能力 无法自我学习 规则需要人工编写
适用范围 简单场景效果好 复杂场景规则爆炸
"规则系统的局限在于：现实世界的知识很难用明确的规则来表达。"
— AI发展史的教训

例子：规则爆炸

假设要写一个判断"这封邮件是不是垃圾邮件"的规则系统：

规则1: 包含"免费" → 可能是垃圾
规则2: 包含"免费"且来自陌生邮箱 → 垃圾
规则3: 包含"免费"且来自陌生邮箱且包含链接 → 垃圾
...
问题：规则越来越多，相互冲突，难以维护

PM的启发

规则系统就像PM制定的"需求规范文档"——明确定义触发条件和处理逻辑。但和规范文档一样，也很难覆盖所有边界情况。

这促使人们思考：能不能让机器自己学习规则？

❌ 失败模式：规则系统的真实代价

案例：某银行风控系统规则爆炸，维护成本失控

背景：2019年，某中型商业银行上线了一套基于规则的风控系统，初期运行良好。

第一年：50条核心规则，覆盖主要欺诈场景
第二年：加到200条规则，业务部门不断追加特殊豁免条件
第三年：规则超过800条，工程师已无法完整梳理所有规则逻辑
问题爆发：
· 规则之间相互冲突，同一用户既被标记"高风险"又被标记"VIP"
· 修改一条规则需要评估对其他20+条规则的影响，平均工时3天
· 每次新业务上线，风控规则开发周期长达2个月
代价：风控团队从3人扩张到15人，年维护成本超过2000万，但风控漏洞率反而上升了40%

规则系统维护成本的具体数字

规则数量 规则冲突概率 单次变更平均工时 维护人力需求
50条以内 低于5% 0.5天 1-2人
100-200条 约15-20% 1-2天 3-5人
500条以上 超过50% 3-7天 10人以上

核心问题：规则系统不是"越全面越好"，当规则数量超过临界点，维护成本会指数级上升，质量反而下降。

✅ 正确姿势

  • 控制规则数量：只把核心、高频、影响大的场景写成规则，其他交给模型
  • 定期清理：每季度review一次规则池，删除已失效或互相冲突的规则
  • 分层设计：简单规则处理高频场景，ML模型处理低频复杂场景
