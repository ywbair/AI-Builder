一、什么是AI Agent（不只是聊天机器人）

聊天机器人 = 问答。AI Agent = 完成任务。

维度聊天机器人AI Agent
交互模式一问一答多步骤执行
目标达成给出回答完成具体任务
自主性被动响应主动规划执行
工具使用不能调用工具、API
典型场景客服、问答自动化流程、数据分析

例子：同一个任务的不同处理方式

任务：帮我分析这个月的销售数据
聊天机器人： "好的，请问您的销售数据在哪里？我可以帮您分析。" AI Agent： 1. 读取Excel文件 2. 自动计算关键指标（营收、增长率、环比） 3. 识别数据异常点 4. 生成分析报告 5. 发送邮件给相关同事

Agent的核心：规划 + 执行 + 反馈循环

Agent不是一次问答就结束，而是循环执行：制定计划 → 执行步骤 → 检查结果 → 调整策略 → 继续执行，直到任务完成。

二、Agent的工作循环

Agent 执行循环

1
感知（Perceive）
理解用户意图，分析当前状态和可用工具
↓
2
规划（Plan）
拆解任务，决定下一步做什么（ReAct模式：推理+行动）
↓
3
执行（Act）
调用工具、API或执行计算
↓
4
评估（Eval）
检查执行结果，判断任务是否完成
↻
"ReAct = Reasoning + Acting"
— ReAct论文（Yao et al., 2022），经典Agent推理框架

例子：Agent如何完成"订机票"任务

用户：帮我订明天北京到上海的机票 Agent循环：
感知：用户要订机票，明天出发，北京→上海
规划：需要先查航班，再选座位，再下单
执行：调用航班查询API
评估：收到3个航班结果，继续规划... 循环继续...
直到完成支付、发送确认邮件

三、Agent容易出现的问题

典型失败模式：错误累积

第1步错10% → 第2步基于错误信息又错10% → 第3步误差越来越大 → 最终结果完全偏离目标

例子：错误累积导致的任务失败

任务：分析竞品A的用户评价 第1步：Agent误把竞品B的数据当成竞品A（错误识别）
↓ 第2步：基于竞品B的数据分析竞品A的用户偏好（错误传递）
↓ 第3步：得出竞品A的结论是"价格敏感"（严重偏差）
↓ 结果：基于错误分析做了错误的定价策略

典型失败模式：死循环

Agent反复执行同一组操作，无法判断任务已完成或无法完成，陷入无限循环。

例子：死循环的场景

任务：在数据库中查找"销售额最高的产品" Agent陷入循环：
执行：SELECT * FROM products ORDER BY sales LIMIT 1
评估：结果看起来不对，再查一次
执行：SELECT * FROM products ORDER BY sales DESC LIMIT 1
评估：还是不对...
执行：SELECT ... （无限重复）

Agent需要"护栏"

  • 最大执行次数限制（防止死循环）
  • 结果验证机制（交叉验证输出的准确性）
  • 异常处理预案（超时时怎么办）
  • 人工介入节点（关键决策需要人确认）

四、RAG：让LLM知道"不知道"的事

LLM的知识有截止日期，不知道实时信息，不知道私有数据。RAG就是解决这个问题的。

RAG 工作流程

1
检索
从知识库找出相关内容
→
2
增强
把检索结果加入Prompt
→
3
生成
LLM基于上下文回答

例子：RAG vs 纯LLM

问题：我们公司2024年的营收目标是多少？
纯LLM回答： "抱歉，我不知道贵公司的具体营收目标..." RAG回答： "根据您公司2024年年度规划文档，贵公司2024年营收目标是3亿元，同比增长30%。"
→ 检索到了内部文档，答案准确

RAG 适用场景

  • 需要回答基于私有文档的问题（合同、制度、文档）
  • 需要引用实时数据（股价、库存、天气）
  • 需要引用特定领域知识（医学、法律、技术）

关键路径：上下文管理是AI应用的核心竞争力

1
LLM 是引擎
决定推理能力上限
2
上下文 是燃料
决定输出质量
3
Agent 是架构
决定能完成什么任务
4
RAG 是外挂
解决知识不足的问题

❌ 失败模式：Agent失控的真实案例

案例：AutoGPT用户让Agent订餐，Agent陷入无限循环并产生高额费用

背景：2023年，一位用户在社交媒体分享了自己使用AutoGPT完成"订餐"任务的经历，教训惨痛。

用户输入："帮我订一份附近评分最高的意大利餐厅的外卖"
Agent开始失控：
· 第1次：读取点评网站API，找到10家餐厅
· 第2次：尝试调用外卖平台API，API返回超时
· 第3次：重试10次API调用 → 触发平台rate limit
· 第4次：换策略，尝试直接打电话给餐厅（没有这个权限）
· 第5次：继续重试，同时生成大量日志文件占用内存
后果：账单高达$1200（大量API调用费用）+ 用户信用卡被冻结 + 餐厅收到数十个无效订单电话
根因：Agent没有"任务完成"的判断能力、缺乏最大执行次数限制、无人工介入节点

Agent失控的典型表现

  • 无限循环：同一组操作反复执行，无法判断任务是否完成
  • 资源耗尽：大量API调用、文件写入、内存占用
  • 越界行为：超出设定权限执行操作（发邮件、发消息、付费）
  • 错误累积：前一步的错误被后续步骤继承并放大
  • 无法自省：Agent意识不到自己陷入了问题

✅ 正确姿势：给Agent加"护栏"

  • 最大执行次数：如循环超过10次则强制停止并报告
  • 预算上限：设置API调用预算，超出则停止
  • 权限边界：明确Agent可以做什么、不可以做什么
  • 人工确认节点：涉及付费、发送等不可逆操作时，必须人工确认
  • 定期checkpoint：每个关键步骤记录状态，出问题时可以回溯
